01 Decision
Decision

Pin one active profile from the tray

Add a Profile submenu to the system tray that lets the operator pin exactly one profile — or stay on Auto (match any), today's behavior. The choice persists in a small state file beside profiles.yaml and is honored by headless runs.

Selection Matching behavior Status
Auto (id "") Match any running player, first profile match wins. Unchanged
Pinned <id> Consider only that profile; lock if its window runs, else stay unlocked. New
"Pin" is a filter, not a force-lock: a pinned profile whose window is not running leaves the adapter unlocked rather than locking nothing-in-particular.
Problem

No way to choose which profile wins

Today the LockManager auto-matches a running player window against all loaded profiles and locks the first match (internal/core/app/lockmanager.go, firstMatch). When several configured players could be running, the operator has no control over which one binds — and no way to force a known-good profile during a live event. The tray exposes only Re-home and Quit.

Goal Pin exactly one active profile from the tray, with Auto restoring match-any.
Constraint Systray-only app (fyne.io/systray), no GUI toolkit. The submenu is the settings page.
Scope was set through brainstorming: pin-one (not enable/disable or force-lock), tray submenu (not web/native UI), separate state file (not mutating profiles.yaml).
Components

Four isolated units

Unit Responsibility Interface
config.SelectionStore
new		 Owns selection.json beside profiles.yaml. Pure I/O — no domain logic, no validity checks. Load() (string, error) · Save(id string) error
LockManager
modified		 Holds a mutex-guarded active string; firstMatch filters to it when set. SetActive(id string) → updates + immediate Poll()
tray.Tray
modified		 Renders the Profile submenu + checkmarks; fires selection callback. onSelectProfile(id string) ("" = Auto)
main.go
wiring		 Loads selection, applies it, wires the callback to Save + SetActive. composition root
The LockManager holds the selection rather than being rebuilt with a filtered profile slice, so the tray callback path stays a single SetActive that both re-matches and re-polls.
Data flow

Startup load, click round-trip

Startup selection.json → Load → LockManager.SetActive → Poll (lock/unlock). In parallel, the tray renders checkmarks from the same id.
On click onSelectProfile(id) → Save(id) → SetActive(id) → Poll → Present. One callback updates persistence, matching, and the displayed lock state. 
startup:  selection.json --Load--> LockManager.SetActive --> Poll (lock/unlock)
                         \--------> tray renders checkmarks

click:    onSelectProfile(id) --> Save(id) --> SetActive(id) --> Poll --> Present
Selection is applied to the LockManager regardless of UI, so headless -no-tray runs honor the persisted file with no menu path.
Implementation draft

Filtered match + re-poll on change

Draft Go — names and signatures may shift against the real types.

// SetActive pins the matcher to one profile (empty id restores match-any),
// then re-polls so the lock state reflects the choice immediately.
func (lm *LockManager) SetActive(id string) {
	lm.mu.Lock()
	lm.active = id
	lm.mu.Unlock()
	lm.Poll()
}

// firstMatch scans profiles, skipping any that are not the pinned one.
func (lm *LockManager) firstMatch(windows []domain.Window) (domain.Profile, domain.Window, bool) {
	lm.mu.Lock()
	active := lm.active
	lm.mu.Unlock()
	for _, p := range lm.profiles {
		if active != "" && p.ID != active {
			continue
		}
		for _, w := range windows {
			if p.MatchesWindow(w) {
				return p, w, true
			}
		}
	}
	return domain.Profile{}, domain.Window{}, false
}
When active == "" the loop is identical to today's path, so existing match-any tests stay green.
Validation

Tests reviewers can scan

Unit Input Expected outcome
SelectionStore Save then Load Round-trips the id.
SelectionStore Missing file ("", nil) — Auto.
SelectionStore Malformed file Returns an error.
LockManager Pin A, only B's window running Stays unlocked.
LockManager Pin A, A and B both running Locks A, never B.
LockManager Active "" Existing match-any tests pass.
LockManager Switch active while locked Re-locks on the triggered Poll.
tray Profiles + selected id (incl. unknown) Resolves the checked entry; unknown → Auto.
Edge cases & operations

Fail loud, fall back to Auto

Stale selection Persisted id no longer in profiles.yaml → log a warning, fall back to Auto. No crash, no silent dead-lock.
Save failure Log the error; the in-memory selection still takes effect this session. Don't block the UI.
Headless -no-tray applies the persisted selection at startup; no interactive change path.
selection.json uses JSON, not YAML, so the app never round-trips and risks the user's commented profiles.yaml.
Sources and caveats

Evidence and out-of-scope

Reviewed lockmanager.go, tray/tray.go, config/store.go, cmd/hyperdeck-adapter/main.go, examples/profiles.yaml.
Out of scope Force-lock override, enable/disable subset, web/native settings UI, a CLI flag to set the selection headless.
Main caveat: radio-style checkmarks are emulated (systray checkboxes are independent), so the tray must clear the others on each pick — covered by the checkmark-resolution test.
